## 基礎
トランジションでラップされた状態更新は裏世界で実行され、レンダリングが完了したら画面に反映される。
トランジション化したものは中断することができる。

以下のような順番。
トランジション状態の更新
→isPendingがtrue
→isPendingによる画面のレンダリング
→この時点でトランジション化した状態更新中ではあるが、トランジションによるレンダリングは裏側で行われており、別の状態更新をできる。（これはつまり状態に優先度が生まれており、トランジションとマークされた状態更新は優先度が低いstate更新と考えて良い。）
→別の状態を更新する
→トランジションによるレンダリングは中断。新しい状態でレンダリングが開始する。

他の例も見てみる。
トランジション状態の更新
→isPendingがtrue
→isPendingによる画面のレンダリング
→この時点でトランジション化した状態更新中
→トランジションのレンダリング完了
→isPendingをfalseにして画面をペイント（トランジション状態の結果を反映したUIが見える）

このように即時UIの更新はされず、isPendingで状態を表現する

## 非同期処理との連携
v19から非同期処理が扱えるようになった。非同期処理が完了するまでisPendingがtrueになる。
また非同期処理の結果を受け取りつつ状態更新をして、それをトランジションするということもできる。

startTransitionとuseReducerによる状態更新を組み合わせたものがuseActionState


## suspenseと連携
suspense内でfetchを行うとサスペンドしてfallbackが表示される。
トランジションを契機にfetchをする場合このfallbackをスキップする
```tsx
const [useId,setUserId] = useState("")

const userProfile = use(dataSource.getUser(userId))

stateTransition(()=>{
  setUserId(input) // ステート更新でサスペンド
})
```
これの良さとしては、例えば初期表示はスケルトンを表示、次のユーザの操作による更新ではスケルトンではなく、別のローディングを表示したい場合、便利。
