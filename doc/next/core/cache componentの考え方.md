## レンダリングタイミング
3種で考える。

1. ラベルのような完全な静的UI(変化することがない)
2．キャッシュできるやつ
3. キャッシュしないやつ

uiが変化するのはなぜか？
それはユーザー固有の情報が必要な場合だ。

つまり、1はユーザ固有の情報ではないので完全にキャッシュ可能でキャッシュした後に変更する必要がない。
したがって1に関してはプリビルドでhtmlを生成する

2に関してはuse cacheディレクティブ、cacheLifeを設定する。 preBuildeされる場合とされない場合の2パターンある。

3はリクエストのたびにレンダリングされる。

cache component以前で実装している場合、dynamicな部分が多く
ラベル一つだけをキャッシュするのか。。。
みたいになっていたがcache componentからは2や3以外が自動的に静的シェルになる。

静的レンダリングは静的シェルと呼ばれ
dynamicの枠組み。みたいになったことで自然な表現になったように感じる

## キャッシュ対象
cache componentではキャッシュする対象が大きく分けて二つ。
関数とページだ。

関数は関数コンポーネントとjsの関数。

以前のキャッシュよりかなり柔軟にできる。

例えばページにつければfull route cacheになり、 preBuilde時にhtmlとなる？

関数コンポーネントであれば、ページの一部のuiがpreBuilde時にhtmlになる？

fetchのラッパー関数につければ、fetchの結果をキャッシュする。これはレンダリング時にリクエストが飛ぶがキャッシュヒットしてすぐにuiが表示される。

## cache componentはデフォルトでdynamic？
キャッシュコンポーネントはデフォルトでdynamicです。
しかし、例えばdynamicの契機となるようなものが一切なく
ページがjsxを返す場合どうなるのか？というと
full route cacheになる

デフォルトでdynamicってことはリクエスト時に毎回作られるのではないの？
となるがそうじゃない。

ここが若干違和感あるが、原因はデフォルトでdynamicの対象。
対象はページ全体のレンダリングではなく、fetch。

ちなみに、静的なページにfetchを書いたときにcache componentは suspenseするか、
use cacheの設定を求めてくる。
つまりエラーに従えば開発者はあまり意識せずに上記の3つを実装することになる。

そもそも、動的な契機がないものはfull route cacheで良いわけで
それを私たちがdynamic force staticとしてマークする必要がないので
cache componenは自然ではある。


## サスペンス
サスペンスでラップすることで、ラップしたものより外側が静的シェルだと教えることができる。
キャッシュをパージしたとき、もしくはキャッシュをしない場合にサスペンスを利用する。
