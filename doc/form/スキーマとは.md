データの構造や制約を定義したもの。

zodのようなスキーマライブラリはスキーマの定義だけでなく型定義、バリデーションロジックを実装できる。

なのでz.objectの戻り値はschemaという命名がふさわしい。

z.inferは名前からわかるようにスキーマから型を推論する
z.inferとz.outputは全く同じで、z.inferはz.outputのエイリアス。（以降outputと呼ぶ）

バリデーションをした結果得られる型定義としてoutputがある。
outputは`.transform()`による型定義変換の影響を受ける。

inputは`.default()`の影響を受ける。
- .default() を付けると、input 側でそのフィールドが optional になる（省略可能になる）
- output 側は常に値が存在する（デフォルト値が補完される）

```tsx
export const blogSchema = z.object() // Zodスキーマオブジェクト。設計図（ランタイムに存在）
export type Blog = z.infer<typeof blogSchema>; // バリデーション済みの実体の型定義。BlogSchemaだとSchemaの型みたいになるのでBlogSchemaという命名はしない
export type BlogItem = Blog & {author_name:string}
```

output=BFFとの契約と考えるとDTOが良いかもしれない。
```tsx
export type CreateBlogItemDTO = z.output<typeof blogSchema>
```

ちなみに、default,transform,preprocessを使っていない場合、input/outputどちらも同じ型定義になる。

ちなみに、preprocessよりもcoerceを使って型を変換してバリデーションすることのほうが多いかも
